# Wireless ADB CLI Plan

## Goal

Build an interactive macOS CLI that wraps Android's wireless debugging flow so the user does not need to remember the underlying `adb` commands.

The tool should:

- Start the wireless pairing flow immediately.
- Guide the user from their Android device to `Developer options -> Wireless debugging`.
- Show an ADB pairing QR code in the terminal.
- Complete ADB pairing and connection after the user scans the QR code.
- Track the device that was just paired so later actions target that device.
- After connection, show a prefixed numbered menu:

```text
airadb> 1. Start scrcpy
airadb> 2. Close
```

The user is assumed to already have developer mode enabled. The CLI must not include instructions for enabling developer mode.

## Research Summary

Primary references:

- Android Developers ADB docs: https://developer.android.com/tools/adb
- AOSP ADB Wi-Fi architecture notes: https://android.googlesource.com/platform/packages/modules/adb/+/HEAD/docs/dev/adb_wifi.md
- ADB man page: https://android.googlesource.com/platform/packages/modules/adb/+/refs/heads/master/docs/user/adb.1.md
- scrcpy connection docs: https://github.com/Genymobile/scrcpy/blob/master/doc/connection.md

Relevant ADB behavior:

- Android 11+ supports wireless debugging through ADB pairing.
- The Mac and Android phone must be on the same Wi-Fi network.
- ADB uses mDNS to discover wireless pairing and connection services.
- Useful commands:

```sh
adb version
adb devices -l
adb mdns check
adb mdns services
adb pair HOST:PORT PAIRING_SECRET
adb connect HOST:PORT
```

Relevant QR behavior:

- Android Studio's QR pairing flow is built on ADB's mDNS pairing service.
- The QR code payload uses this shape:

```text
WIFI:T:ADB;S:studio-<random-instance>;P:<pairing-secret>;;
```

- `T:ADB` tells Android this is an ADB wireless debugging QR code.
- `S:` requests the mDNS pairing service instance name, usually a random `studio-*` value.
- `P:` carries the pairing secret generated by the host.
- After the phone scans the QR, it starts an `_adb-tls-pairing._tcp` mDNS service using the requested instance name.
- The CLI can poll `adb mdns services`, find that service, then run `adb pair <ip>:<port> <pairing-secret>`.
- After pairing succeeds, ADB normally auto-discovers `_adb-tls-connect._tcp`. If it does not, the CLI should use `adb connect <ip>:<port>` as a fallback when it can identify the connection endpoint.

Relevant scrcpy behavior:

- `scrcpy` uses ADB to communicate with the device.
- The CLI should launch scrcpy with the serial of the device that was just paired.
- Match the Raycast extension configuration shown in the reference screenshot:
  - Device Serial: use the just-paired serial.
  - Screen Size: device size, so do not pass `--max-size`.
  - Disable audio: enabled, so pass `--no-audio`.
  - Turn screen off: disabled, so do not pass `--turn-screen-off`.
  - Stay awake: enabled, so pass `--stay-awake`.
  - HID keyboard: disabled, so do not pass `--keyboard=uhid`.
  - HID mouse: disabled, so do not pass `--mouse=uhid`.
  - Always on top: enabled, so pass `--always-on-top`.
  - Audio codec: `opus`. Because audio is disabled, do not pass `--audio-codec=opus` in the default command; `opus` is scrcpy's default codec if audio is enabled later.
- Effective default launch command:

```sh
scrcpy -s <serial> --no-audio --stay-awake --always-on-top
```

For a wireless ADB device, the serial may be an `ip:port` address or an mDNS-style ADB serial from `adb devices`. Passing the serial avoids accidentally opening scrcpy for an unrelated device that was already connected.

## Recommended Stack

Use Rust for the CLI.

Reasons:

- Produces a single native binary for macOS.
- Good fit for interactive terminal UX.
- Easy to spawn `adb` and `scrcpy` without shell quoting hazards.
- Strong enough parsing and error handling without a runtime dependency.

Suggested crates:

- `clap` for CLI flags.
- `anyhow` or `thiserror` for errors.
- `dialoguer` or `inquire` for prompts.
- `qrcode` plus terminal rendering, or `qr2term`, for displaying the QR code.
- `rand` for generating the `studio-*` instance name and pairing secret.
- `regex` for parsing `adb devices -l` and `adb mdns services`.
- `which` for checking required binaries.

## CLI Shape

Initial command:

```sh
airadb
```

Optional flags:

```sh
airadb --adb /path/to/adb
airadb --scrcpy /path/to/scrcpy
airadb --timeout 90
airadb --no-scrcpy-check
```

The MVP can implement only the default interactive command. Flags can be added after the first working flow if desired.

## User Flow

1. Start the CLI.

```text
airadb> Checking ADB...
```

2. Validate prerequisites.

- `adb` exists.
- `adb version` works.
- `adb mdns check` works, or prints a useful warning.
- `scrcpy` is checked lazily when the user chooses menu option `1`.

3. Start QR pairing.

Do not run a user-visible device detection or device selection step before pairing. A first-time wireless phone will not appear in `adb devices -l`, and existing ADB devices should not change the main path. The startup flow should always guide the user to scan a fresh pairing QR code.

Show Android-side instructions.

Keep this short and assume developer mode is already enabled:

```text
airadb> On your Android phone:
airadb> Go to Developer options -> Wireless debugging.
airadb> Tap "Pair device with QR code".
airadb> Scan the QR code below.
```

Do not tell the user how to enable developer mode.

Generate QR pairing details.

- Generate a service instance name:

```text
studio-<10 random safe chars>
```

- Generate a pairing secret, for example 12 to 20 characters from an alphanumeric alphabet.
- Avoid characters that need Wi-Fi QR escaping in the MVP.
- Build:

```text
WIFI:T:ADB;S:<service-instance>;P:<pairing-secret>;;
```

- Render this as a terminal QR code.

4. Wait for the phone to scan the QR.

At this point the phone is still not expected to show up in `adb devices -l`. The QR scan causes Android to advertise an ADB pairing service over mDNS. The CLI should watch mDNS, not the device list.

Poll:

```sh
adb mdns services
```

Look for a row where:

- service name equals the generated `studio-*` instance, and
- service type is `_adb-tls-pairing._tcp`.

Extract `HOST:PORT`.

5. Complete pairing.

Run without a shell:

```sh
adb pair <host>:<port> <pairing-secret>
```

Expected success output includes a successful pairing message. On failure, show the command's stderr/stdout and offer to generate a new QR code.

6. Connect to the just-paired device.

Only after `adb pair` succeeds should the CLI expect the phone to become a ready ADB device. The CLI should target the phone that just scanned the QR code, not any unrelated device that was already connected.

Discover the wireless debugging connection endpoint:

```sh
adb mdns services
```

Prefer an `_adb-tls-connect._tcp` service on the same host/IP as the pairing endpoint. When exactly one plausible connection endpoint is found, run:

```sh
adb connect <host>:<port>
```

Record that `<host>:<port>` as the selected serial for the rest of the session.

Then verify the device is ready:

Poll:

```sh
adb devices -l
```

Find the ready `device` entry that matches the recorded wireless serial. If `adb connect` reports a different serial or ADB normalizes the serial differently, use that value.

If the CLI cannot identify the just-paired device, show a clear error and offer to retry QR pairing. Do not ask the user to choose among unrelated existing devices.

7. Show the post-connection menu.

```text
airadb> Connected to <serial> <model-if-known>
airadb> 1. Start scrcpy
airadb> 2. Close
airadb> Select an option:
```

Option `1`:

- Check `scrcpy` exists.
- Start scrcpy against the just-paired device serial:

```sh
scrcpy -s <paired-serial> --no-audio --stay-awake --always-on-top
```

- Keep the CLI alive while `scrcpy` is running so the user can see errors.

Option `2`:

- Exit cleanly.
- Do not disconnect ADB automatically in the MVP unless the user explicitly asks for that in a later feature.

## Internal Modules

Suggested structure:

```text
src/
  main.rs
  adb.rs
  qr.rs
  ui.rs
  scrcpy.rs
```

`adb.rs`

- Resolve `adb` path.
- Run `adb` commands.
- Parse `adb devices -l`.
- Parse `adb mdns services`.
- Pair with a discovered pairing endpoint.
- Connect to a discovered connection endpoint.

`qr.rs`

- Generate safe random service instance names.
- Generate safe random pairing secrets.
- Build ADB QR payloads.
- Render terminal QR codes.

`ui.rs`

- Print prefixed status lines.
- Render numbered menus.
- Read and validate numeric input.
- Keep Android-side instructions concise.

`scrcpy.rs`

- Resolve `scrcpy` path.
- Build the default Raycast-equivalent argument list:
  - `-s <paired-serial>`
  - `--no-audio`
  - `--stay-awake`
  - `--always-on-top`
- Launch `scrcpy` with that argument list.

## Parsing Notes

`adb devices -l` example shape:

```text
List of devices attached
R5CT123ABC device product:foo model:Pixel_8 device:bar transport_id:1
192.168.1.50:42131 device product:foo model:Pixel_8 device:bar
```

Data model:

```rust
struct AdbDevice {
    serial: String,
    state: DeviceState,
    product: Option<String>,
    model: Option<String>,
    device: Option<String>,
    transport_id: Option<String>,
}
```

`adb mdns services` example shape:

```text
List of discovered mdns services
studio-AbCdEf1234 _adb-tls-pairing._tcp 192.168.1.23:37199
adb-XYZ-QXjCrW _adb-tls-connect._tcp 192.168.1.23:40233
```

Data model:

```rust
struct MdnsService {
    instance: String,
    service_type: String,
    address: String,
}
```

## Error Handling

Missing `adb`:

```text
airadb> adb was not found. Install Android SDK Platform Tools and make sure adb is on PATH.
```

Missing `scrcpy` when option `1` is selected:

```text
airadb> scrcpy was not found. Install scrcpy, then choose option 1 again.
```

mDNS unavailable:

- Print `adb mdns check` output.
- Explain that QR pairing depends on ADB mDNS discovery.
- Suggest making sure the Mac and phone are on the same Wi-Fi network.
- Suggest trying a non-corporate or less restricted Wi-Fi network if discovery never appears.

Unauthorized USB device:

```text
airadb> <serial> is visible but unauthorized. Unlock the phone and approve the debugging prompt, then retry.
```

Offline device:

```text
airadb> <serial> is offline. Toggle Wireless debugging off/on, then retry.
```

Pairing timeout:

- Let the user retry with a new QR code.
- Let the user close the tool.

Ambiguous connection endpoint:

- Do not ask the user to choose from already-connected devices.
- Prefer the `_adb-tls-connect._tcp` endpoint on the same host/IP as the QR pairing endpoint.
- If there is still more than one plausible endpoint, explain that the just-paired phone could not be identified and offer to retry QR pairing.

## Acceptance Criteria

MVP is done when:

- Running `airadb` checks for `adb`.
- The tool starts QR pairing immediately after prerequisite checks.
- The tool does not ask the user to select from existing ADB devices.
- The terminal displays a scannable ADB QR code.
- The Android instructions say only:
  - go to `Developer options -> Wireless debugging`,
  - tap `Pair device with QR code`,
  - scan the QR code.
- After QR scan, the tool finds the generated `_adb-tls-pairing._tcp` service.
- The tool runs `adb pair <host>:<port> <secret>`.
- The tool does not expect an unpaired wireless phone to appear in `adb devices -l`.
- The tool uses mDNS and `adb connect` to target the just-paired phone.
- The tool verifies the just-paired phone with `adb devices -l` only after QR scan and successful `adb pair`.
- The tool shows:

```text
airadb> 1. Start scrcpy
airadb> 2. Close
```

- Entering `1` starts `scrcpy -s <paired-serial> --no-audio --stay-awake --always-on-top`.
- Entering `2` exits cleanly.

## Test Plan

Unit tests:

- Parse `adb devices -l` with:
  - no devices,
  - one USB device,
  - one wireless device,
  - multiple devices,
  - unauthorized/offline devices,
  - models containing underscores.
- Confirm existing devices do not trigger a startup selection flow.
- Parse `adb mdns services` with:
  - generated `studio-*` pairing service,
  - unrelated pairing services,
  - `_adb-tls-connect._tcp` services,
  - empty service list.
- Generate QR payloads with safe characters.
- Validate numeric menu input.
- Build the default scrcpy argument list from the Raycast-equivalent profile.

Integration tests with fake binaries:

- Put fake `adb` and `scrcpy` scripts earlier in `PATH`.
- Simulate command outputs over time:
  - no device -> QR scan -> pair success -> connected device,
  - existing unrelated device -> QR scan -> pair success -> just-paired device selected,
  - pair failure,
  - mDNS timeout,
  - multiple connect endpoints where only one matches the pairing host,
  - ambiguous connect endpoints where retry is offered,
  - scrcpy missing,
  - scrcpy launch failure,
  - scrcpy receives `-s <paired-serial> --no-audio --stay-awake --always-on-top`.

Manual macOS test:

1. Install Android SDK Platform Tools and scrcpy.
2. Run `airadb`.
3. On Android, go to `Developer options -> Wireless debugging`.
4. Tap `Pair device with QR code`.
5. Scan the CLI QR code.
6. Confirm the CLI reports the connected phone.
7. Choose `1` and confirm scrcpy starts.
8. Close scrcpy, rerun `airadb`, and confirm it shows a fresh QR pairing flow rather than an existing-device selection.
9. Test with two phones or one phone plus an emulator already connected and confirm scrcpy still targets the phone that scanned the QR code.
